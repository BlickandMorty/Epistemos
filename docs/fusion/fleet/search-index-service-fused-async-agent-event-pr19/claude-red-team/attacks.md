## Attacks

### A1 — `offloadSearch` operation closure is non-async; @MainActor recorder cannot be `await`-ed inside it [P0]
**Surface:** Implementation contract bullet "Emit rows only after `normalizedSearchTerms(query)` succeeds and before the existing `dbPool.read` RRF execution starts." Files: `Epistemos/Sync/SearchIndexService.swift:888-924` (`offloadSearch`), `Epistemos/Sync/SearchIndexService.swift:591-626` (`fusedSearchAsync`), `Epistemos/Engine/AgentToolProvenanceRecorder.swift:1-22` (recorder is `@MainActor`, `Persist` typealias is `@MainActor (AgentProvenanceEvent) -> Bool`).
**Attack:** PR18's ShadowSearchService can `await agentProvenanceRecorder.recordToolEvent(...)` because its `search` method is an async actor method — recorder calls happen in actor context that can cross to `@MainActor`. SearchIndexService's `fusedSearchAsync` puts the body inside `offloadSearch`'s `@Sendable @escaping (OffloadedSearchCancellationProbe) throws -> T` operation closure (lines 596 + 889). That closure is **non-async** and is dispatched onto `queryQueue.async` — `await` is impossible inside it. The brief therefore cannot be implemented by copying the PR16-PR18 pattern: `requested`/`started` rows literally cannot be emitted from "after `normalizedSearchTerms` succeeds and before `dbPool.read` starts" the way the contract says. Implementer's only options are (a) hoist recorder calls out of `offloadSearch` (emits "started" before the workItem actually runs, opening a queue-wait window where started is recorded but no work has begun), (b) refactor `offloadSearch` to take an `async` operation closure (breaks `searchAsync` and `searchBlocksAsync` shape), or (c) `Task { await ... }` fire-and-forget (no ordering guarantees, completed can race past failed).
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:888-924`, `Epistemos/Sync/SearchIndexService.swift:596` (`try await offloadSearch { [self] cancellation in`), `Epistemos/Engine/AgentToolProvenanceRecorder.swift:3-9`, `Epistemos/Engine/ShadowSearchService.swift:74-91` (PR18 pattern that does NOT go through an offload-style closure).
**Mitigation proposed:** Rewrite the contract to explicitly authorize the actor-scope-then-offload split: emit `requested` + `started` on the SearchIndexService actor (or `@MainActor`) **immediately before** the `try await offloadSearch { ... }` call; emit `completed`/`failed` immediately after, threading the elapsed time and a Sendable success/failure summary out of the offload closure. Brief must also call out that "started" is the actor-scope point of dispatch, not the queue-execution start. Otherwise Kimi will copy the ShadowSearch pattern verbatim and the build will fail.

### A2 — Default-value `AgentToolProvenanceRecorder` forces SearchIndexService.init onto `@MainActor`, breaking nonisolated production callers [P0]
**Surface:** Implementation contract bullet "Add `AgentToolProvenanceRecorder` injection to `SearchIndexService` using the existing PR16-PR18 pattern." Files: `Epistemos/Sync/SearchIndexService.swift:137` (`init(databaseURL providedDatabaseURL: URL? = nil) throws`), `Epistemos/Engine/AgentToolProvenanceRecorder.swift:11-22` (`@MainActor` initializer).
**Attack:** ShadowSearchService dodged this by marking both inits `@MainActor` (lines 36-46 of `ShadowSearchService.swift`). SearchIndexService's `init` is **not** `@MainActor` today, and forbidden-files list blocks edits to `Epistemos/Sync/VaultSyncService.swift` and any other call site. If PR19 either (a) marks `SearchIndexService.init` `@MainActor` to allow a default-arg recorder, that breaks every existing nonisolated synchronous caller (`VaultSyncService`, `AppBootstrap`, tests), all of which are protected; or (b) makes the recorder argument optional and lazy-initializes it inside the actor on first use, it then has to construct a `@MainActor` recorder from a non-`@MainActor` actor context — which requires a `MainActor.assumeIsolated` (deadlock-prone if the call originates on the main thread already, given the canonical CLAUDE.md "DispatchQueue.main.async in UniFFI callbacks, NEVER .sync (deadlock)" rule). The brief authorizes neither path explicitly.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:137`, `Epistemos/Engine/ShadowSearchService.swift:36-46`, deliberation §"Forbidden files/subsystems" listing `Epistemos/Sync/VaultSyncService.swift`. CLAUDE.md (root) "DispatchQueue.main.async in UniFFI callbacks, NEVER .sync (deadlock)" — same hazard with `MainActor.assumeIsolated` from a non-actor caller.
**Mitigation proposed:** Add an explicit contract bullet: "Recorder is injected via a new `AgentToolProvenanceRecorder?` argument with `nil` default; when `nil`, a `@MainActor` recorder is constructed lazily on first use via `Task { @MainActor in ... }` and stored in actor state. SearchIndexService.init remains nonisolated and existing callers do not need to know about the recorder." Verify this lazy path is reachable from the `actor` context by hand-tracing the first `await` on `fusedSearchAsync`.

### A3 — Cancellation terminal-row contract is unsatisfiable in three observable cases [P0]
**Surface:** Acceptance bullet "cancellation records one terminal failed row with `failure_class=cancelled`" plus implementation contract bullet "Preserve `fusedSearchAsync` ... cancellation behavior". Files: `Epistemos/Sync/SearchIndexService.swift:888-924` (`offloadSearch`), `Epistemos/Sync/SearchIndexService.swift:893-894` (pre-checkCancellation).
**Attack:** Three race windows produce zero or two terminal rows, contradicting "exactly one":
1. **Pre-task-cancellation:** line 894 calls `try Task.checkCancellation()` **before** the continuation is even installed. If the parent Task is already cancelled at entry, `fusedSearchAsync` throws CancellationError immediately, no `requested`/`started`/`failed` rows are ever produced. Acceptance says cancellation produces one terminal row — this case produces zero. The acceptance test will be flaky depending on when the test cancels.
2. **Post-`bind`/pre-`workItem-run` cancellation:** if `onCancel` fires after `currentState.bind(workItem:)` (line 917) but before `queryQueue.async(execute:)` actually pulls the work item off the queue, the workItem is cancelled (line 89), `currentState.cancel()` resumes the continuation with CancellationError — but the `requested`/`started` rows that A1's hoisted-emission solution would have put **before** `offloadSearch` are already in the recorder. Result: one started row, one failed row, no completed — OK if the contract authorizes it, but the brief says "Preserve ... cancellation behavior" which today emits zero events.
3. **Post-success cancellation:** the workItem completes successfully (calls `currentState.finish(with: .success(...))`) on the queue thread; concurrently `onCancel` fires `stateBox.cancel()`. The `Mutex<Storage>.completed` flag in `OffloadedSearchState` (lines 56-67 + 76-90) makes the second `finish` a no-op, so the continuation resumes successfully. But if recorder calls are emitted from inside the workItem (A1 path c) AND from `onCancel`, both `completed` AND `failed` rows can land for a single call. Acceptance says exactly one terminal row.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:893-894`, `:899-918`, `:75-91`, `:103-108`. Acceptance §3 "cancellation records one terminal failed row".
**Mitigation proposed:** Either (a) explicitly authorize "cancellation produces ≤ 1 terminal row" (best effort) and adjust the acceptance test to tolerate the pre-Task.checkCancellation zero-row case; or (b) move `requested`/`started` emission to **before** the `try Task.checkCancellation()` so even a pre-cancelled Task records two events plus a synthetic terminal `failed`/`cancelled` written from the actor scope; or (c) rewrite the cancellation surface to a single emission point owned by the actor that observes `OffloadedSearchState.cancelled` and emits exactly one terminal row at the end of `fusedSearchAsync`. Brief must pick one and write the test against it.

### A4 — GRDB DatabaseError messages contain user query fragments; unconstrained `failure_class` derivation can leak query bytes into AgentEvent metadata [P1]
**Surface:** Implementation contract "bounded failure class" + forbidden grep `localizedDescription|String(describing:.*error|sanitized`. Files: `Epistemos/Sync/SearchIndexService.swift:583-585` (existing catch path), `Epistemos/Sync/RRFFusionQuery.swift:43` (`SearchFusionMetrics`).
**Attack:** GRDB's `DatabaseError.message` for an FTS5 syntax error contains the bad fragment verbatim — e.g. a query like `foo "bar` becomes `unrecognized token: "\"bar"` in `error.localizedDescription`. The brief says "bounded failure class" but does not enumerate the closed set, does not specify a sanitization function, and does not block sub-string sniffing. An implementer who writes `if message.contains("syntax") { .invalid_input }` is technically reading user query bytes inside the recorder code path. Worse: even if `errorMessage` is set to the slug only, if the implementer logs `log.warning("fused search failed: \(error)")` (PR18 does this with `.public` privacy at `ShadowSearchService.swift:150`), the GRDB error string with the raw user fragment lands in OSLog with `privacy: .public`. PR19 is not allowed to do that under MAS. The forbidden grep blocks `localizedDescription`, but does not block plain `\(error)` interpolation in os_log.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:583-585`, `Epistemos/Engine/ShadowSearchService.swift:150` (precedent that the brief's "PR16-PR18 pattern" includes a `.public` raw-error log line), CLAUDE.md (root) "REAL APIs ONLY ... HONEST CAPABILITY GATING".
**Mitigation proposed:** Add a contract bullet enumerating the closed `failure_class` set explicitly — `cancelled | sql_error | unknown_error` is sufficient for an FTS5 path; do NOT inherit ShadowSearch's `not_found` or `rust_panic` (no Rust here) or `invalid_input` (already screened by `normalizedSearchTerms`). Add a forbidden grep `(log\.|os_log|Logger).*\\\\(error` against `Epistemos/Sync/SearchIndexService.swift` for the PR19 diff. Mandate that failure-class derivation switches on the **type** of error (`is GRDB.DatabaseError`, `is CancellationError`) and never inspects the message field.

### A5 — Sync `fusedSearch` accidental instrumentation passes the positive-grep guards [P1]
**Surface:** Acceptance bullet "sync `fusedSearch` remains uninstrumented" + failure-proof guardrails (positive greps only). Files: `Epistemos/Sync/SearchIndexService.swift:557-587` (sync `fusedSearch`), `Epistemos/Sync/SearchIndexService.swift:591-626` (async `fusedSearchAsync`).
**Attack:** All three failure-proof guardrails are positive: `search-index-fused-async-`, `toolName: "search_index.fused_search_async"`, `surface": "fused_search_async"`. None of them assert that the **sync** method `fusedSearch(...)` does not contain `recordToolEvent` or `agentProvenanceRecorder`. An implementer who refactors a shared helper into both methods would silently start emitting AgentEvents for the sync path too — useful for coverage, but breaks the "sync remains uninstrumented" acceptance, and breaks the "actor/MainActor recorder hop" privacy budget because the sync method is `nonisolated public` and called from `QueryRuntime.fullText` and other hot paths.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:557` (`nonisolated public func fusedSearch`), Forbidden files list (the sync method is in an allowed file but the contract forbids its instrumentation).
**Mitigation proposed:** Add a negative grep to the failure-proof guardrails: `git grep -n -A 30 "nonisolated public func fusedSearch(" Epistemos/Sync/SearchIndexService.swift | grep -E "(agentProvenanceRecorder|recordToolEvent)"` returns empty. Add a green test that calls sync `fusedSearch` with an injected recorder sink and asserts `sink.recordedEvents.isEmpty`.

### A6 — Throws preservation requires recording terminal `failed` before re-throw, with bounded recorder latency [P1]
**Surface:** Implementation contract bullet "Preserve `fusedSearchAsync` throws behavior, `SearchFusionMetrics`, signposts ...". Files: `Epistemos/Sync/SearchIndexService.swift:621-624` (existing catch), `Epistemos/Engine/AgentToolProvenanceRecorder.swift:25-75` (recorder is synchronous-ish: returns `Bool`, but `await` from non-MainActor is real wall-time).
**Attack:** The current catch block does `SearchFusionMetrics.shared.recordError(error); throw error`. If PR19 inserts `await recordTerminalFailedRow(...)` between the `recordError` and the `throw`, then the recorder must complete before the caller observes the thrown error. If the recorder is slow (MainActor saturated, EventStore.shared mid-WAL-checkpoint), the throw is delayed — that's a behavior change for callers like `QueryRuntime.fullText` that may have cancellation deadlines. If the implementer instead uses `Task { @MainActor in await recorder.record... }` + immediate re-throw, the terminal row is fire-and-forget and may land **after** the next `requested` row from the same SearchIndexService instance, breaking sequence ordering and confusing replay tooling.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:583-585` + `:621-624`, `Epistemos/Engine/AgentToolProvenanceRecorder.swift:50-74` (`sequenceByRunID` is shared mutable state, so out-of-order `record` calls give monotonic sequences within a run but the wall-clock occurredAtMs may be inverted vs. start of the next run on the same recorder instance).
**Mitigation proposed:** Mandate `await` the terminal recorder call **before** `throw` and assert in a green test that the AgentEvent for the failed row has a strictly-earlier `occurredAtMs` than any subsequent `requested` row from the same SearchIndexService. Forbid the `Task { @MainActor in }` fire-and-forget pattern explicitly.

### A7 — Tool-call-ID monotonic counter ownership unspecified — concurrent `fusedSearchAsync` calls collide [P1]
**Surface:** Implementation contract "tool ids `search-index-fused-async:N`". Files: `Epistemos/Engine/ShadowSearchService.swift:34` (`searchSequence: UInt64`), `Epistemos/Sync/SearchIndexService.swift:557` (`fusedSearch` is **nonisolated** — not actor state safe), `:591` (`fusedSearchAsync` IS actor-isolated).
**Attack:** ShadowSearch puts `searchSequence` on the actor — safe because all callers go through the actor's serial executor. SearchIndexService's `fusedSearchAsync` is actor-isolated (good), but the brief never specifies the counter is actor state. If the implementer mirrors `SearchFusionMetrics.shared` (line 44 of `RRFFusionQuery.swift`) and makes the counter a static singleton with NSLock, sequences are global — fine. If the implementer makes it per-instance but stores it in a `nonisolated(unsafe) var` for symmetry with `fusedSearch`, two concurrent `fusedSearchAsync` calls (legal, given offloadSearch dispatches to a `.concurrent` `queryQueue` at line 162) get duplicate tool-call IDs from incrementing the same memory non-atomically. Test for "monotonic per-instance" must include concurrent-call coverage.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:159-163` (`queryQueue` is `attributes: .concurrent`), `:591` (`fusedSearchAsync` not nonisolated), `Epistemos/Engine/ShadowSearchService.swift:34`.
**Mitigation proposed:** Add a contract bullet: "Tool-call-ID counter is actor-isolated state on SearchIndexService; sequence is per-instance and monotonic." Add a green test that fires N=8 concurrent `fusedSearchAsync` calls against an injected recorder and asserts the captured `tool_call_id`s form a strict permutation of `1...8` with no duplicates.

### A8 — Empty-after-sanitization queries (e.g. `&&`, all stopwords, all emoji) silently produce zero events but pass acceptance test for "valid hit" [P1]
**Surface:** Acceptance bullet "Green tests prove valid zero-hit searches complete with `hit_count = 0`" vs. "Green tests prove empty/whitespace queries emit no events". Files: `Epistemos/Sync/SearchIndexService.swift:601-602` (`guard !terms.isEmpty else { return [] }`).
**Attack:** `normalizedSearchTerms("&&")` and `normalizedSearchTerms("🤖🤖🤖")` plausibly normalize to `[]` (FTS5 unicode61 tokenizer drops symbol-only inputs). The current code returns `[]` silently. The contract says "Emit rows only after `normalizedSearchTerms(query)` succeeds" — succeeds is ambiguous: is non-empty-output success, or is non-throwing success? An input that passes the user-facing "non-whitespace" check but normalizes to an empty term list silently produces zero events AND zero hits AND consumes no `dbPool.read`. The acceptance test for "valid zero-hit search" probably uses a real word like `nonexistentword`, so the post-normalization-empty case is untested. A sanitizer rewrite later (PR20+) that changes `normalizedSearchTerms` semantics could silently reroute these queries through the recorder, surprising consumers.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:601-602`, deliberation §"Acceptance" bullets 1-4.
**Mitigation proposed:** Tighten the contract: "Emit rows iff `normalizedSearchTerms(query)` returns non-empty AND parent Task is not already cancelled. Empty-after-normalization is treated identically to empty-input: no events." Add a green test for input `"&&"` asserting zero events.

### A9 — Pre-Task-cancellation produces zero events; acceptance "cancellation records one terminal failed row" is flaky [P1]
**Surface:** Acceptance bullet 3 "cancellation records one terminal failed row with `failure_class=cancelled`". Files: `Epistemos/Sync/SearchIndexService.swift:893-894` (`try Task.checkCancellation()` before continuation install).
**Attack:** Same root as A3 but written for the test author. If the test does `let task = Task { try await service.fusedSearchAsync(...) }; task.cancel(); await task.value` — Swift schedulers may elect to never schedule the task body, so `Task.checkCancellation()` at line 894 throws and zero rows land. The test will pass on a cold runner and fail on a warm one. The brief must specify the cancellation injection point: cancel the Task **after** awaiting a synchronization point that proves `requested` has been recorded, or use a recorder sink that returns a continuation the test resumes after observing the requested row.
**Evidence:** `Epistemos/Sync/SearchIndexService.swift:893-894`, `Epistemos/Sync/SearchIndexService.swift:888-924`.
**Mitigation proposed:** Add a contract bullet: "Cancellation-emits-terminal-row green test must use a recorder sink with a `recordedHook: @Sendable (event) -> Void` that signals an `AsyncStream` after the `started` row lands; the test cancels the Task only after consuming that signal." Mandate the test name `fusedSearchAsync records exactly one terminal failed row on mid-flight cancellation` (no ambiguity about pre-flight cancellation).

### A13 — Persisted `weights` are scalar today but can leak query intent if a future caller passes user-tunable weights [P1]
**Surface:** Implementation contract "Persist only scalar sanitized metadata: ... weights ...". Files: `Epistemos/Sync/RRFFusionQuery.swift` (`FusionWeights` struct), `Epistemos/Sync/VaultSyncService.swift:2326` (current consumer).
**Attack:** `FusionWeights` today is `.default` for every caller, so persisting the four scalars is harmless. The persisted weights then become an indirect timestamp / intent signal in the AgentEvent ledger, surviving export and replay forever. Brief must either restrict persistence to "weights == .default ? 'default' : 'custom'" (one bit), or whitelist the four `.default` scalars and reject unknown weight shapes.
**Evidence:** `Epistemos/Sync/RRFFusionQuery.swift:43` comment ("ring buffer for ... observability"), `Epistemos/Sync/VaultSyncService.swift:2326`, `MASTER_RESEARCH_INDEX_2026_05_02.md §5`.

Add a green test that calls `fusedSearchAsync(weights: .default)` and asserts the captured `argumentsJSON` contains only the literal string `"default"` for the weights field.

## Brief verdict

The brief is well-scoped — one-file additive provenance over an already-bounded async chokepoint, correct anchors, correct forbidden list. But it inherits the "PR16-PR18 pattern" by reference, and that pattern does not transplant to `SearchIndexService.fusedSearchAsync` because the async wrapper goes through `offloadSearch`'s non-async closure and SearchIndexService's init is not `@MainActor`. The smallest revision that closes all P0/P1 attacks is:

1. Pick A1's mitigation (a) — hoist recorder calls out of `offloadSearch`, emit `requested`/`started` in the actor scope before the await, emit `completed`/`failed` in the actor scope after — and write that explicitly into the contract.
2. Pick A2's mitigation — `nil`-default optional recorder argument with lazy `@MainActor` construction.
3. Pick A3 mitigation (a) or (b) — explicit "≤ 1 terminal row" semantics for cancellation, with the test using a synchronization point per A9.
4. Add A4's `cancelled | sql_error | unknown_error` closed set + a `(log\.|os_log|Logger).*\\\\(error` forbidden grep.
5. Add A5's negative-grep guardrail and sync-uninstrumented green test.
6. Add A11's `Gate: SovereignGate touchpoint? none` field.
7. Tighten run-id (A12) and weights (A13) format strings.
8. Add A10's existing-test migration bullet and A14's UNIFIED_SUBSTRATE update bullet.

With those eight bullets folded in, Kimi can ship the slice in one revision. Verdict: **brief-revise**, not blocked — the structural contradiction in A1 is the only true gate.
