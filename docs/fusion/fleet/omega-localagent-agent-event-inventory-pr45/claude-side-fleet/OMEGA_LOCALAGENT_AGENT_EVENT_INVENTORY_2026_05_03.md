## 3. No-instrument decisions

These surfaces should NOT emit AgentEvents. Each entry has a one-line reason; the full pattern is in `AGENT_EVENT_RUNTIME_COVERAGE_MAP_2026_05_03.md` §4.

- **Routers** (`HybridRouter`, `DualBrainRouter`, `ConfidenceRouter`, `FallbackChainResolver`): routing decisions are not tool dispatch. Recording them as tool events would conflate planning with action.
- **Parsers** (`ToolCallParser`, `IncrementalToolCallDetector`, `LocalToolGrammar`, `ToolSchemaGrammar`): parsing model output into structured tool calls is interpretation, not execution. Tool events fire at the dispatch site (LocalAgentLoop, Bridge/*, AgentQueryEngine), not the parser.
- **Inference engines** (`MLXConstrainedGenerator`, `ConstrainedDecodingService`, `DeviceAgentService`): the model itself isn't a tool. Token streaming has its own observability via `StreamingDelegate`. Treating inference as a tool event would 100×–10000× the AgentEvent table.
- **Policy lookups** (`HermesGatewayPolicy`, `AgentAuthority`, `ToolSurfacePolicy`): pure decision functions. The decision is consumed by an upstream dispatcher that already emits the event.
- **State / config** (`OrchestratorState` (stub), `ChannelRegistryState`, `HardwareTierManager`, `AgentBackend`, `AgentHandoff`, `AgentUsageLedger`, `ChatCapability`, `OmegaPermissions`, `TCCPermissionState`): containers and types, not actions.
- **Vision helpers covered upstream** (`Screen2AXFusion`, `AXorcistBridge`, `ScreenCaptureService`, `AXMutationDetector`, `VisualVerifyLoop`): all callers already instrumented at the Bridge layer (PR39 + PR40 + PR41 + PR42). Adding rows here would double-count.
- **Internal safety / accounting** (`CredentialRedactor`, `MemoryThreatScanner`, `TranscriptRepair`, `MMRReranker`, `ContextCompiler`, `ContextBudgetManager`, `CostTracker`, `ToolLoopDetector`, `AgentDepthLimiter`): these run BESIDE tool execution as guards. They aren't user-invoked actions.
- **Knowledge helpers** (`GraphDataModel`, `RecipeGraphSkills`, `GhostBrainCoauthor`, `ODIATraceGenerator`, `TraceDataMixer`): internal indexers / mixers / coauthors. The user doesn't dispatch these as tools; the upstream LocalAgentLoop / AgentQueryEngine already record what they trigger.
- **Diagnostics** (`IMessageNativeSetupDoctor`, `IMessageReplyDelegate`): covered by `IMessageDriverService.swift` (instrumented) or are pure setup probes.
- **Stub** (`OrchestratorState.swift`): file's own header reads "The full Omega orchestrator has been retired in favor of the Rust agent_core. This stub preserves the public API surface…". Don't instrument a stub.

The full no-instrument list also covers the four Bridge files from the original coverage-map §4: `StreamingDelegate.swift`, `ChunkedMCPFraming.swift`, `CoTStreamInterceptor.swift`, `ToolTierBridge.swift`.

## 4. Conflicts / uncertainties

A handful of surfaces are ambiguous from current source alone. Each needs a Codex detective pass before a code slice opens. **Do not** open a PR for any of these without resolving the question.

### C-1 — Is `GhostComputerAgent.swift` on a currently-reachable production path?

- **Question:** does any production caller still reach `GhostComputerAgent.execute(step:)` after `ComputerUseBridge` (PR39) became the canonical computer-use entrypoint?
- **Why it matters:** if NO, GhostComputerAgent is dead code superseded by `ComputerUseBridge`; the right action is to delete the file (separate cleanup card), not instrument it. If YES, Gap-1 (§2) is real and the slice should open.
- **Suggested detective method:** grep for `GhostComputerAgent(`, `OmegaAgent` instantiation sites, and `ComputerUseBridge` callers; trace which one(s) `AgentViewModel` and `ChatCoordinator` actually invoke today. Also check `AppBootstrap.swift` for which agent instances are constructed at runtime.

### C-2 — Is `Epistemos/Omega/Knowledge/AgentGraphMemory.swift` upstream-covered?

- **Question:** when `recordExecution` and `distillMemory` write to the agent graph memory, is the upstream caller (LocalAgentLoop / AgentQueryEngine / ReasoningLoopService) already recording an AgentEvent that captures the same logical write?
- **Why it matters:** if upstream callers already cover it, AgentGraphMemory is `no-instrument` (would double-count). If they don't, this is a real gap (memory writes are MEDIUM payload risk — execution metadata can include sensitive partial chat content).
- **Suggested detective method:** trace each call site of `AgentGraphMemory.recordExecution(...)` and `AgentGraphMemory.distillMemory(...)`; check whether the surrounding code emits `recordToolEvent` with `.toolCallCompleted` for the same logical operation.

### C-3 — Is `Epistemos/Omega/Safety/ShadowGitCheckpoint.swift` user-tool-exposed?

- **Question:** is `ShadowGitCheckpoint.checkpoint(...)` / `rollback(...)` callable from the agent as a user-facing tool, or is it strictly a background safety mechanism that the user never directly triggers?
- **Why it matters:** background safety mechanisms are `no-instrument` (own observability via OS log / signpost is sufficient). User-facing tools need a `recordToolEvent` row so the audit timeline includes the rollback action.
- **Suggested detective method:** grep for callers of `ShadowGitCheckpoint.shared.checkpoint(...)` / `.rollback(...)`; confirm whether any of them are wired to a tool name in the registry vs. invoked from internal lifecycle hooks.

### C-4 — Should the `Epistemos/Omega/Vision/VisualVerifyLoop.swift` emit lifecycle events?

- **Question:** the visual verify loop is invoked AFTER a computer-use action to confirm the action took effect. PR39 (ComputerUseBridge) records the action. Does the verify loop run as part of the same logical action (covered by PR39) or as a separate step (would need its own row)?
- **Why it matters:** if it runs inside the same Bridge call (covered), it's `no-instrument`. If it runs as a separate post-action audit pass, it deserves its own `verify.completed/failed` event so failed-verify cases are visible in the audit timeline.
- **Suggested detective method:** check `ComputerUseBridge.execute(actionJSON:)` to see whether it calls `VisualVerifyLoop` directly, and whether the recorded AgentEvent for the bridge action includes verify-result metadata.

### C-5 — Are there reasoning rounds in `ReasoningLoopService.swift` that bypass the existing PR7 instrumentation?

- **Question:** PR7 instrumented internal think → critique → tool-use rounds, but `ReasoningLoopService.swift` is large (per `ReasoningRound` shape). Are there code paths (early-exit, error-path, fallback) that don't hit the instrumented emit site?
- **Why it matters:** if early-exit paths drop the `completed` event, audit timelines show `requested` rows with no terminal state — the same anti-pattern as Gap-2 (LocalAgent reflex/EOF).
- **Suggested detective method:** trace every `return` from `ReasoningLoopService.runRound(...)` and confirm each one fires through the recorder.
