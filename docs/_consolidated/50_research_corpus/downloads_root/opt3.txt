Architectural Transformation of the Epistemos Source Engine for ProMotion Performance: A Technical Blueprint for 120fps Code Editing
The quest to deliver a code editor capable of maintaining a consistent 120fps refresh rate on ProMotion displays—specifically within the context of the Epistemos editor and its CodeEditSourceEditor foundation—requires a radical departure from standard high-level abstractions in favor of a specialized, hardware-accelerated rendering and data management pipeline. For modern macOS systems, the performance ceiling is no longer defined by raw CPU clock speed but by the efficient orchestration of the main thread, the effective use of background concurrency, and the direct utilization of GPU bandwidth.1 To match the fluidity of Xcode or industry-leading native editors like Zed, the Epistemos engine must overcome three primary categories of bottlenecks: the    complexity of SwiftUI’s declarative state management, the synchronous nature of syntax highlighting queries, and the CPU-bound limitations of CoreText-based rasterization.3
The Performance Paradox of Declarative Text Systems
The current Epistemos implementation utilizes a SwiftUI SourceEditor view that relies on a Binding<String> to handle text content. While this pattern aligns with SwiftUI’s state-driven philosophy, it introduces a catastrophic performance penalty in the context of high-frequency text editing. In a declarative framework, a modification to a bound string triggers a re-evaluation of the entire view tree or, at the very least, a diffing operation that compares the previous version of the string with the current one.3
The Computational Cost of SwiftUI String Diffing
When a user types a single character in a 10,000-line file, the $text: Binding<String> mechanism forces the framework to treat the entire file as a value type. This necessitates a full-string comparison to identify the delta for update propagation.7 In documents of significant size, this    diffing process consumes a substantial portion of the frame budget. On a 120Hz ProMotion display, the total time allotted for a single frame is approximately 8.33 milliseconds.
  

If the string diffing and subsequent view re-rendering exceed this 8.33ms window, the system experiences a "hitch," leading to dropped frames and perceived stuttering.3 Research into SwiftUI performance reveals that even small view updates can trigger re-evaluations of every row in a list or every line in a text container if the state is not properly segregated.6
Transitioning to Reference-Based Text Storage
The resolution to this bottleneck lies in bypassing SwiftUI’s value-type diffing by adopting a reference-based model using NSTextStorage. Unlike a standard Swift String, NSTextStorage is a semi-concrete subclass of NSMutableAttributedString that maintains a persistent backing store and notifies observers only of specific, localized changes.10 By wrapping the editor in an NSViewRepresentable and maintaining a single reference to the NSTextStorage object, the editor can perform in-place mutations that do not trigger a global SwiftUI re-render.5

High-Concurrency Syntax Highlighting and Tree-Sitter Optimization
Syntax highlighting in Epistemos is driven by Tree-sitter, a high-performance incremental parsing system. However, the current implementation often experiences "choppy" highlighting because query execution and attribute application are frequently tied to the main thread's run loop.12
Parallelizing the Parsing Pipeline
Tree-sitter’s primary advantage is its ability to reuse unchanged subtrees during a re-parse, reducing the computational cost of an edit from a full-file parse to a localized update.14 Despite this efficiency, the process of querying the resulting Abstract Syntax Tree (AST) for highlight captures can still be time-consuming for complex grammars or files with high density.14
To achieve 120fps, the syntax highlighting must move to a fully asynchronous model. The TreeSitterClient should operate on a dedicated serial background queue or within a Swift 6 Actor to ensure that the main thread is never blocked by parsing or query logic.14 Because Tree-sitter cursors and parsers are not thread-safe, the system must maintain separate instances or strictly serialize access.14
Viewport-Aware Querying and Lazy Highlighting
A significant optimization is the transition from full-file highlighting to viewport-aware querying. The editor should only request highlights for the range of text currently visible on the screen, plus a small buffer to account for scrolling.14 The SwiftTreeSitterLayer abstraction facilitates this by allowing queries to be restricted to specific NSRange bounds, thereby minimizing the number of nodes the query engine must traverse.16
When an edit occurs, the following sequence should be executed:
1. Edit Notification: The NSTextStorage notifies the TreeSitterClient of the edited range.
2. Incremental Parse: The client updates the existing AST using ts_tree_edit and performs a localized re-parse.14
3. Background Query: A query for highlight captures is executed in the background for the current visible range.
4. Asynchronous Application: Highlight attributes are batched and applied to the NSTextStorage on the main thread during the next available idle period.16
This "lazy" approach ensures that even in massive files, the work performed by the syntax highlighter is strictly bounded by the screen real estate, not the document length.14
GPU-Driven Text Rendering: The Path to 120Hz Fluidity
The most profound bottleneck in current macOS text editors is the reliance on CPU-bound rasterization via Core Graphics and CoreText. Standard NSTextView architectures involve the CPU calculating glyph positions and then drawing those glyphs into a bitmap context, which is then uploaded to the GPU as a texture for display.17 This process is inefficient for high-refresh-rate scrolling because even small movements require the CPU to re-calculate and re-rasterize entire line fragments.
The Metal Glyph Atlas Architecture
To achieve the performance characteristic of the Zed editor or Xcode’s optimized layers, Epistemos must adopt a GPU-driven rendering pipeline. This architecture treats text rendering like a video game: the CPU performs the initial glyph shaping (using CoreText to determine glyph indices and positions), but the actual drawing is handled by the GPU using a "Glyph Atlas".1
A Glyph Atlas is a pre-rendered GPU texture containing all common characters of a font. Instead of rasterizing strings, the editor generates a vertex buffer where each character is represented by a quad (two triangles). These quads point to specific UV coordinates in the atlas texture.4

	By moving the drawing to Metal, the editor becomes GPU-bandwidth-bound rather than CPU-bound. In this model, scrolling a file of any size is virtually "free" for the CPU, as the GPU can render millions of quads in a single draw call with negligible latency.4
Multi-Channel Signed Distance Fields (MSDF)
To ensure that text remains crisp across the dynamic scaling and zoom levels characteristic of a modern macOS interface, the Glyph Atlas should utilize Multi-Channel Signed Distance Fields (MSDF). Unlike standard bitmap atlases, which become blurry when magnified, MSDF encodes distance information in the RGB channels of the texture.4
The fragment shader in the Metal pipeline uses this distance data to reconstruct sharp edges at any scale. The median of the RGB channels provides the signed distance, and the fwidth function allows the shader to apply an adaptive smoothing ramp that matches the screen's pixel density exactly.4 This ensures that even at 120Hz, every character remains perfectly sharp, with zero rasterization artifacts during rapid motion.
Triple Buffering and Sync Management
A common issue in Metal-based macOS applications is the introduction of "hitches" due to synchronization bottlenecks between the CPU and GPU. Zed’s development team discovered that while M1 chips typically operate in "composited mode," newer M2 and M3 chips often run in "direct mode," which can cause the main thread to block on waitUntilCompleted calls for significantly longer than expected.2
To mitigate this, the Epistemos Metal pipeline must implement triple buffering of its instance buffers. This involves maintaining a pool of buffers; while the GPU is reading from Buffer A to render Frame 1, the CPU can be writing to Buffer B to prepare Frame 2.2 Furthermore, replacing waitUntilCompleted with wait_until_scheduled ensures that the application can begin processing the next frame as soon as the GPU has been successfully instructed, rather than waiting for the entire rendering process to finish.2
Peripheral Subsystems: Minimaps and Gutters
Secondary UI elements such as the minimap and the gutter are often overlooked in performance profiles, yet they frequently account for significant CPU overhead during scrolling.12
CALayer Caching for Minimap Rendering
The minimap in CodeEditSourceEditor currently suffers from high re-rendering costs.12 A high-performance minimap should not be a "live" text view. Instead, it should be a specialized CALayer that captures a downsampled representation of the document. By setting shouldRasterize = true on the minimap's backing layer, the macOS render server can cache the visual state of the minimap as a texture.21
During a scroll event, the minimap only needs to update the "visible window" indicator. The actual text content of the minimap remains static on the GPU, requiring zero CPU cycles to redraw.21 The cache only needs to be invalidated when the underlying text changes, significantly reducing the burden on the main thread during navigation.
Metal-Accelerated Gutter and Folding Ribbon
Similarly, the gutter (containing line numbers and folding markers) can be integrated into the Metal draw call. Rather than using an array of NSView or NSTextField objects—which are expensive to manage in files with 100,000 lines—the gutter should be rendered as a single set of quads within the same Metal command buffer as the text.20 This unified rendering approach ensures that the gutter and the source code are perfectly synchronized during 120fps scrolling, eliminating the "lagging gutter" effect often seen in less optimized editors.
Advanced Display Synchronization on Apple Silicon
Achieving 120fps is not merely about rendering quickly; it is about rendering at the precise moment the display is ready to refresh.
CVDisplayLink and the 8.33ms Deadline
On macOS, CVDisplayLink is the optimal API for synchronizing a custom Metal rendering loop with the hardware's V-sync.24 Unlike CADisplayLink, which is sometimes capped at 60Hz by the OS to save power, CVDisplayLink provides high-precision callbacks that reflect the true physical refresh rate of the ProMotion display.2
The Epistemos engine should use CVDisplayLink to drive its entire update-render loop. Every 8.33ms, the link triggers a callback that must:
1. Process Input: Handle keyboard or mouse events.
2. Update State: Calculate new cursor positions or scroll offsets.
3. Prepare Mesh: Generate the Metal vertex buffers for the visible viewport.
4. Encode Commands: Send the draw calls to the GPU.
5. Present: Commit the command buffer with a present_drawable call.2
Preventing ProMotion Downclocking
A subtle challenge with ProMotion displays is their tendency to "downclock" to 24Hz or 30Hz when the screen is static to conserve battery.2 This can lead to a sluggish feel when a user begins to scroll after a period of inactivity. To solve this, the editor should adopt the Zed strategy of "rendering repeated frames" for a short duration (typically 500ms to 1000ms) after any user interaction.2 This signal tells the macOS window server that the application requires maximum responsiveness, effectively locking the display at 120Hz during the entire editing session.
Detailed Integration Strategy for Epistemos
The transition from the current CodeEditSourceEditor architecture to an optimized engine involves a systematic replacement of the core text-handling and rendering layers.
Step 1: Replace Binding with Reference-Based State
The first priority is the elimination of the $text binding in CodeEditorView.swift. A new SourceEditorState class should be introduced to act as the single source of truth, owning the NSTextStorage instance.

   let treeSitterClient: TreeSitterClient
   
   init(initialText: String) {
       self.textStorage = NSTextStorage(string: initialText)

This state is then passed to the NSViewRepresentable, which manages the CodeEditTextView lifecycle.5
Step 2: Implement Background Highlighting with Neon
The TreeSitterClient must be modified to perform all parsing and query operations within a serial background context. Integration with the Neon library is recommended to manage the complex task of applying high-frequency highlight updates to the NSTextStorage without blocking the main thread.16
Step 3: Deploy the Metal Rendering Layer
The TextView must be overhauled to use a CAMetalLayer as its backing store. The current CoreText-based drawRect logic should be migrated to a Metal vertex-buffer generator that utilizes an MSDF glyph atlas for text rendering.4
Step 4: Synchronize via CVDisplayLink
Finally, the entire editor must be tethered to a CVDisplayLink to ensure that every frame is delivered in sync with the ProMotion refresh cycle. This includes the implementation of triple-buffering for Metal instance buffers to ensure maximum throughput on Apple Silicon.2
