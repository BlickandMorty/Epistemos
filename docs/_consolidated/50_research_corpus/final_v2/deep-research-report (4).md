## 🧠 Part 1: Moats & North Star Audit  
- **Data & Privacy Moat:** Epistemos’s local vault of user knowledge is a classic **data moat**.  As Cuofano notes, “Only moats that compound with usage survive”【22†L25-L30】. Every note, search, and tag strengthens Epistemos’s unique dataset. This personal knowledge graph cannot easily be replicated by competitors, protecting user lock-in.  
- **Local-AI Moat:** Epistemos plans to run **on-device LLMs** (Qwen3, Apple Foundation Models) and caching (Metal shaders, quantization). This focus on local inference (no sidecar) means low latency and offline use. It also ties into **privacy** and **offline-first** moats: user data never leaves the machine, which users may prefer over cloud-only apps.  
- **Ecosystem Moat:** Tight integration with macOS (Swift UI, Keychain, ScreenCaptureKit) and Swift/Rust tech is a niche moat. Being optimized for Apple’s unified memory and APIs (AXUI, MLX-Swift) makes Epistemos stickier for Mac users. Even Apple’s keyboard focus hints (proactive suggestions, etc.) could be leveraged as a platform moat.  
- **Agentic & Workflow Moat:** Epistemos’s “Hermes agent” orchestration, bridging Claude/Codex-style cloud LLMs with local models and tools, can create an **ecosystem of plugins**. A well-designed skills system and multi-step planning (as in the research docs) could become a moat if it lets users automate complex tasks within Epistemos’s vault. 

🔍 *North Star*: A suggested North Star metric would tie to *customer value*. For a PKM app, a strong candidate is something like **“Meaningful Knowledge Created per Active User.”** This could count new notes, links, or AI-driven summaries that lead to insights. As Amplitude writes, the North Star should capture the **value delivered**【23†L179-L188】. For example, measuring how often a user successfully finds answers in their vault, or builds new knowledge structures, aligns the product vision with growth (team alignment, reduced waste【23†L179-L188】). That North Star then drives development trade-offs: features should aim to boost genuine user insights, not vanity metrics.  

## 🧠 Part 2: Concept & Implementation Research  
We now examine each major concept/feature from the project docs, summarizing purpose, approach options, and evidence.  

- **Structured Output (“Grammar Masking”)**: Epistemos plans to enforce LLM outputs against a schema. Apple’s new `@Generable` macro (iOS 26) exemplifies this: define a Swift `struct` as the schema and the model will output valid JSON matching it【18†L27-L36】. For example, a `struct Reminder: Decodable {...}` forces the LLM to output exactly those fields. This yields **guaranteed type-safe** responses. Using prompts like `/nothink` (instructing no extra text) further constrains output【19†L1-L4】. In practice, Rudrank Riyam shows combining MLX Swift and `@Generable` yields robust JSON (field-by-field streaming in SwiftUI)【18†L27-L36】【19†L1-L4】. **Tradeoff:** this requires the model to understand the schema; errors can occur if it breaks format. But it greatly simplifies parsing. *Recommendation:* integrate MLX Swift’s structured generation (maybe using `MLX-Swift-Structured` library) and test on our use cases, as the groundwork is largely present in `LocalToolGrammar.swift`.  

- **KV-Cache Quantization**: On Apple’s unified memory Macs, KV cache size is a bottleneck【8†L62-L69】. Two leading ideas:  
  - **KIVI (2-bit asymmetric)**: A 2024 ICML paper (Liu et al.) introduces KIVI, which quantizes keys per-channel and values per-token to 2 bits. It yields ~2.6× reduction in peak memory and ~2.35–3.47× throughput gains【12†L67-L74】. KIVI is “tuning-free” and hardware-friendly. Implementing KIVI means carefully following its scheme (arXiv details key/value quantization separately).  
  - **TurboQuant (3-bit)**: Google’s ICLR 2026 TurboQuant further compresses KV from FP16 to 3-bit with near-zero loss【8†L62-L69】. An Apple-optimized implementation achieved ~5× KV compression on large models (Gemma 26B, Qwen3 9B)【8†L62-L69】. A Swift port (`turboquant-mlx`) exists and may be drop-in.  
  - **Option Comparison**: KIVI’s 2-bit is more aggressive (4× memory cut) but may introduce more quantization error than TurboQuant’s 3-bit, though KIVI’s design reportedly keeps quality almost intact【12†L67-L74】. On Apple hardware, initial tests suggest int4 (4-bit) already allows extreme context lengths【9†L404-L413】. The best approach might be: implement KIVI as an *opt-in flag* (as mentioned in docs) and benchmark it. Also track TurboQuant code (arozanov’s repo) – if stable, it could replace a custom solution【8†L62-L69】.  

- **UniFFI (Rust↔Swift FFI)**: The project uses Mozilla’s UniFFI to bind Rust core logic to Swift. Moving from v0.28 to 0.29.5 involves **breaking changes**: 
  - `UniffiCustomTypeConverter` is **removed** (replaced by `custom_type!` macro)【6†L102-L110】. Any old Rust impls must switch.  
  - External types annotations change: `[External=".."] typedef extern` → `[External=".."] typedef enum/interface` or use `[Custom=..]`【6†L139-L147】【6†L169-L177】.  
  - Also, v0.30 changed method-checksum rules, so pinning to 0.29.5 exactly is wise (as docs warn【6†L102-L110】).  
  In summary, update bindings with new macros, remove old traits, and keep uniffi-bump in one PR. Check Swift output errors carefully. Since UniFFI is fundamental, we should follow Mozilla’s upgrade guide【6†L102-L110】.  

- **B-Tree Rope Text Buffer**: For large multi-megabyte notes and efficient edits, a rope data structure is ideal. Many Rust editors (Helix, xi, Lapce) use ropes. Recent work (“crop” crate) shows a B-tree rope is **very fast**【26†L88-L94】. In practice, using a UTF-8 B-tree rope (like `crop` or `ropey`) means insertions/deletions scale logarithmically, and **cheap snapshots** (for undo or multi-thread) are trivial. The docs plan a “rope” crate plus Swift UniFFI bindings. This is sound: we should use an existing Rust rope (e.g. [crop](https://github.com/noib3/crop)【26†L88-L94】) and expose it to Swift. This allows fast text operations and O(1) snapshots, as the docs note.  

- **Operation Log (OpLog)**: Epistemos intends an append-only **single-writer operation log** (feature-flagged). This is akin to *event sourcing*: every edit or action is logged, enabling time travel and easy sync (since only one writer). The benefit is simplicity: unlike full CRDT automerge, a single-writer log just appends. Implementation: maintain a Rust-backed log of operations (graph mutations, text edits). On re-open or sync, replay the log. We should ensure the log format is compact (e.g. deltas) and immutability helps. No direct citation, but this approach is common in distributed systems for *auditability*. It fits the priority: with typestate (next item) managing pointers, OpLog yields cheap snapshots (the “rope provides O(1) snapshots for OpLog” as docs suggest).  

- **Typestate Pattern in Rust**: The docs mention “Typestate Islands” requiring that honest FFI pointers be typestates. In Rust, the **typestate pattern** encodes valid states in the type system, preventing invalid usage at compile time【24†L5-L12】. For example, a resource handle might only allow `read()` when in an “Open” state. Cliffle’s article shows using typestate prevents bugs by making illegal states unrepresentable【24†L5-L12】. Epistemos’s plan (doing honest FFI first, then typestate) suggests they’ll use typestate to ensure, say, once a session is closed you can’t resume it. We should research crates like `type-state` or use newtype wrappers. This will add ~650 LOC but greatly improve safety. **Note:** typestate code can be verbose; require documentation and tests.  

- **Background Jobs / ETL (Apalis)**: For tasks like crawling external knowledge (ETL), the docs propose Apalis (Rust background job library)【29†L95-L100】. Apalis is designed for simple multi-threaded jobs【29†L95-L100】. This fits: we can dispatch crawl jobs as Apalis tasks, possibly storing results back to the vault. Apalis supports queues, Redis backend, etc. We should test Apalis with our workload (web scraping, repo scanning). If heavy I/O is needed, Apalis with Tokio should handle it. The docs pin version `=1.0.0-rc.7`, so ensure compatibility.  

- **Other UI & Utilities**: Many tasks (cost dashboard, vault selector, etc.) are straightforward SwiftUI views; these are “90% done” per docs. Using SwiftUI and Combine should suffice. Keychain usage is mandated (secure API keys)【2†L198-L202】. Apple’s documentation recommends Keychain for secrets【2†L198-L202】. Ensure no sensitive logging. For example, using `Keychain` APIs or libraries in SwiftUI is standard practice【30†L5-L8】.  

## ⚡ Synthesis & Insights  
- **Memory & Performance**: A strong theme is **work within Apple Silicon’s memory**. Many topics (KIVI/TurboQuant, bit-packed circuits, no-copy buffers) aim to cram models and context into ~11GB. Using 4-bit quantized models (Qwen3, Gemini) and 2–3bit KV caches is key. The research suggests *we can push context beyond 100K tokens* if quantized (see Open-TQ results【9†L404-L413】). This is critical for Epistemos’s vision of infinite context.  
- **Structured vs Unstructured**: The move toward grammar/schema-based output is clear. By constraining LLMs, we avoid brittle parsing. It’s a trend Apple enables on-device【18†L27-L36】, and it aligns with the app’s need for tool-like interactions (making reminders, notes, code snippets automatically).  
- **Framework Trends**: The tech choices (SwiftUI, Metal, MLX-Swift, Rust+UniFFI) reflect a strategy: **use official tools for stability**. For example, MLX-Swift’s `@Generable` macro means we’re on Apple’s supported path for on-device AI【18†L27-L36】.  
- **Risk & Gaps**: Some items are high-risk (Bucket D). E.g. personalized embeddings (W9.11) sounds innovative but a large learning curve. We would need to research Apple’s Create ML surfaces, which lack documentation. Or thermal throttling (W9.29) might require deep understanding of SMC and low-level APIs. For now, focus on concrete tasks (Buckets A–C) and consider D as future experiments.  

### 📝 Recommended Next Steps  
- **Moat Validation**: Talk to potential users about what they value (privacy? offline search? speed?), to refine the North Star metric. Possibly survey how many insights users gain or tasks they accomplish with Epistemos.  
- **Prototype Key Features**: Build a minimal structured-output flow (Swift struct + prompt) and test it with MLX to ensure it meets real use cases (e.g. summarizing or task creation).  
- **Benchmark Quantization**: Using the `apozanov/turboquant-mlx` repo and possibly a KIVI implementation, run benchmarks on our target hardware (simulate heavy context).  
- **Design OpLog Format**: Define the schema for the operation log (JSON or binary?). Perhaps reuse SQLite (since GRDB is in the stack) to append ops, with indexing by block or time.  
- **Prepare UniFFI Migration**: Audit current Rust→Swift bindings for Uniffi usage. Identify where `custom_type!` needs to replace old converters. Automate binding regeneration cautiously.
